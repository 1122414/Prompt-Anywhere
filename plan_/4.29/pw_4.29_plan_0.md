# Prompt Anywhere 4.29 增强计划：模板变量 + Composer（最小 UI 改动版）

## 0. 本轮目标

基于当前 Prompt Anywhere 已有 UI 和功能，在**不大幅修改系统页面**、**不破坏现有功能**的前提下，新增两类能力：

```txt
1. 模板变量：把普通自然语言提示词逐步转成可填空模板。
2. Composer：把多个已有提示词按顺序拼接成一个完整提示词。
```

本轮不是重做 UI，不是重构架构，不是接入 AI。

核心原则：

```txt
当前三栏主界面继续保留。
当前目录树继续保留。
当前搜索结果弹窗继续保留。
当前编辑 / 渲染模式继续保留。
当前复制、保存、导出、导入、批量管理、收藏、最近使用继续可用。
新增功能只通过少量按钮、右键菜单、弹窗实现。
```

## 5. 功能一：模板变量

### 5.1 功能定位

模板变量不是要求用户手写 `{{变量名}}`，而是允许用户把普通提示词中“以后可能变化的部分”转成变量。

核心流程：

```txt
用户粘贴普通提示词
↓
用户选中某段文本
↓
右键“转为变量”
↓
输入变量名
↓
系统自动替换为 {{变量名}}
↓
之后点击“使用模板”
↓
弹出变量填写表单
↓
生成最终提示词
↓
复制 / 预览 / 导出
```

### 5.2 变量语法

统一使用：

```txt
{{变量名}}
```

变量名规则：

```txt
1. 只能使用英文、数字、下划线。
2. 必须以英文字母或下划线开头。
3. 不允许空变量名。
4. 同一个变量名可在文件中出现多次。
5. 使用模板时，同名变量只填写一次。
```

合法示例：

```txt
{{project_name}}
{{company}}
{{target_role}}
{{output_format}}
{{requirements}}
```

非法示例：

```txt
{{项目名称}}
{{1name}}
{{project-name}}
{{ }}
```

非法时必须提示用户修改，不要静默失败。

### 5.3 UI 增量：编辑器右键菜单

在当前编辑模式的文本编辑区域中，新增右键菜单项：

```txt
转为变量
查看变量
使用模板
```

#### 转为变量

仅当用户选中文本时可用。

流程：

```txt
选中文本
↓
右键：转为变量
↓
弹出“变量名输入框”
↓
输入变量名
↓
确认
↓
将选中文本替换为 {{变量名}}
```

示例：

```txt
原文：
请帮我优化 AutoBrowser 项目经历。

选中：
AutoBrowser

变量名：
project_name

结果：
请帮我优化 {{project_name}} 项目经历。
```

#### 查看变量

扫描当前文件正文，列出所有变量：

```txt
project_name
company
output_format
```

如果没有变量：

```txt
当前提示词中没有模板变量。
```

#### 使用模板

如果当前文件包含变量，弹出变量填写窗口。

如果当前文件不包含变量：

```txt
当前提示词没有模板变量，将直接复制原文。
```

### 5.4 UI 增量：顶部按钮区

在现有按钮区中新增两个按钮，但不要破坏原布局：

```txt
[使用模板]
[组合器]
```

建议位置：

```txt
复制 / 保存 / 导出 / 删除 附近
```

如果按钮过多，可以放到一个“更多”菜单中：

```txt
更多
├── 使用模板
├── 查看变量
├── 打开组合器
└── 加入组合器
```

### 5.5 变量输入弹窗

弹窗名称：

```txt
使用模板
```

内容：

```txt
文件名：当前提示词文件名

变量表单：
project_name:  [                    ]
company:       [                    ]
requirements:  [ 多行文本框          ]
output_format: [                    ]

下方按钮：
[生成预览] [复制结果] [导出结果] [取消]
```

要求：

```txt
1. 支持单行输入。
2. 支持多行输入。
3. 生成预览后不修改原始文件。
4. 复制结果复制的是替换后的最终文本。
5. 导出结果导出的是替换后的最终文本。
6. 用户取消时不改变文件内容。
```

### 5.6 模板替换规则

实现一个独立服务：

```txt
TemplateService
```

职责：

```txt
1. extract_variables(content) -> list[str]
2. validate_variable_name(name) -> bool
3. replace_selection_with_variable(content, selection_range, variable_name) -> str
4. render_template(content, values) -> str
```

替换规则：

```txt
1. 找到所有 {{变量名}}。
2. 用用户填写值替换。
3. 同名变量全部替换。
4. 未填写变量时阻止生成，并提示用户填写。
```

### 5.7 变量名建议

变量输入弹窗中提供常用变量名快捷按钮：

```txt
project_name
company
target_role
task
requirements
constraints
output_format
tone
length
code
text
background
acceptance_criteria
forbidden
```

用户点击后自动填入变量名。

注意：

```txt
这只是快捷填充，不是 AI 自动识别。
```

### 5.8 模板变量验收标准

```txt
[ ] 用户选中文本后可以右键转为变量。
[ ] 未选中文本时“转为变量”不可用或提示需要先选中文本。
[ ] 非法变量名会被阻止。
[ ] 当前文件可以扫描出所有 {{变量名}}。
[ ] 点击“使用模板”可以弹出变量填写表单。
[ ] 填写变量后可以生成预览。
[ ] 复制结果是替换后的最终文本。
[ ] 原始 .md / .txt 文件不会被替换结果覆盖。
[ ] 同名变量只需要填写一次。
[ ] 当前提示词没有变量时，使用模板不会报错。
```

## 6. 功能二：内置模板库

### 6.1 功能定位

内置模板用于降低用户学习成本。

本轮可以加入少量默认模板，但仍然使用本地 `.md` 文件，不做数据库。

### 6.2 内置模板目录

目录由 `.env` 控制：

```env
BUILTIN_TEMPLATE_DIR=./builtin_templates
```

建议结构：

```txt
builtin_templates/
├── 开发计划/
│   └── opencode开发计划.md
├── 代码/
│   ├── 代码审查.md
│   └── 代码精简重构.md
├── 简历面试/
│   ├── 简历项目润色.md
│   └── 面试口述回答.md
└── 文档/
    └── 文档润色.md
```

这些模板本质仍然是 `.md` 文件。

### 6.3 导入内置模板

提供入口：

```txt
导入内置模板
```

可以放在：

```txt
顶部“导入文件 / 导入文件夹”旁边
或
设置按钮菜单中
或
更多菜单中
```

流程：

```txt
点击导入内置模板
↓
弹出模板列表
↓
用户选择一个或多个
↓
选择导入到哪个分类
↓
复制到 data/分类/
```

### 6.4 内置模板示例

#### opencode 开发计划模板

```md
请为项目 {{project_name}} 生成一份详细可执行的开发计划。

当前问题：
{{current_problem}}

本轮目标：
{{goal}}

必须实现：
{{requirements}}

严禁实现：
{{forbidden}}

验收标准：
{{acceptance_criteria}}

要求：
1. 按优先级拆分任务。
2. 每个任务写清楚边界。
3. 每个任务写清楚验收方式。
4. 明确哪些配置放入 .env。
5. 输出为可直接保存的 plan.md。
```

#### 简历项目润色模板

```md
请帮我优化下面的简历项目经历。

目标岗位：{{target_role}}
项目名称：{{project_name}}
技术栈：{{tech_stack}}
希望突出：{{highlight}}
不希望出现：{{avoid}}

原始项目描述：
{{raw_description}}

要求：
1. 不要使用没有量化支撑的虚词。
2. 尽量使用可解释、可防守的指标。
3. 控制在 {{length_limit}} 字以内。
4. 输出中文简历项目 bullet。
```

### 6.5 验收标准

```txt
[ ] 可以从 builtin_templates 读取模板。
[ ] 可以选择模板导入到 data 目录。
[ ] 导入后的模板能在目录树中显示。
[ ] 导入后的模板能使用模板变量功能。
[ ] 内置模板不影响用户已有文件。
```

## 7. 功能三：Composer 组合器

### 7.1 功能定位

Composer 用于把多个已有提示词文件拼接成一个完整提示词。

它解决的是：

```txt
全局规则 + 任务模板 + 输出格式 + 当前需求
```

这类需要多段提示词组合的问题。

### 7.2 最小 UI 方案

不改主界面布局，新增一个独立弹窗：

```txt
Composer 组合器
```

打开方式：

```txt
1. 顶部按钮：[组合器]
2. 文件右键菜单：加入组合器
3. 搜索结果右键菜单：加入组合器
4. 当前打开文件按钮：加入组合器
```

Composer 弹窗可以是：

```txt
┌──────────────────────────────────────────────┐
│ Composer 组合器                               │
├──────────────┬──────────────┬────────────────┤
│ 可选提示词     │ 已选择片段     │ 组合预览        │
│              │              │                │
│ Coding       │ 1. 全局规则    │ # 全局规则      │
│ 面试          │ 2. 代码精简    │ ...            │
│ 简历          │ 3. 当前任务    │ ---            │
│              │              │ # 代码精简      │
├──────────────┴──────────────┴────────────────┤
│ [加入] [上移] [下移] [移除] [清空]              │
│ [复制组合结果] [使用模板变量] [导出 md] [保存组合] │
└──────────────────────────────────────────────┘
```

### 7.3 Composer 基础功能

必须实现：

```txt
1. 从已有提示词文件中选择多个文件。
2. 将文件加入组合区。
3. 在组合区调整顺序。
4. 从组合区移除文件。
5. 清空组合区。
6. 自动生成组合预览。
7. 一键复制组合结果。
8. 导出组合结果为 .md。
9. 保存组合结果为新的 .md 文件。
```

### 7.4 Composer 拼接规则

默认拼接格式：

```md
# 文件名去后缀

文件内容

---
```

示例：

```md
# 全局规则

你是一名严谨的代码助手。

---

# 代码精简规则

优先删除、合并、复用现有代码。

---

# 当前任务

请基于以下需求生成计划。
```

规则从 `.env` 读取：

```env
COMPOSER_SEPARATOR=\n\n---\n\n
COMPOSER_INCLUDE_FILE_TITLE=true
COMPOSER_TITLE_LEVEL=1
```

如果：

```env
COMPOSER_INCLUDE_FILE_TITLE=false
```

则只拼接内容。

### 7.5 Composer 与模板变量结合

Composer 生成组合结果后，需要检测组合结果中是否存在 `{{变量名}}`。

如果存在：

```txt
显示按钮：[使用模板变量]
```

点击后：

```txt
组合结果
↓
扫描变量
↓
弹出变量填写表单
↓
生成最终组合结果
↓
复制 / 导出
```

注意：

```txt
使用模板变量不会修改参与组合的原始文件。
```

### 7.6 保存组合

保存组合有两种方式。

#### 保存组合结果

把拼接后的文本保存为普通 `.md` 文件。

路径：

```txt
data/组合模板/xxx.md
```

目录从 `.env` 读取：

```env
COMPOSER_SAVE_DIR=./data/组合模板
```

#### 保存组合配置

本轮可选，不强制。

如果要做，保存为 JSON 到 app_state：

```json
{
  "name": "opencode开发计划组合",
  "files": [
    "data/全局规则/AGENTS护栏.md",
    "data/Coding/代码精简.md",
    "data/模板/opencode开发计划.md"
  ]
}
```

本轮建议优先做“保存组合结果”，不要一开始做复杂组合配置管理。

### 7.7 Composer 搜索复用

Composer 中的可选提示词列表可以复用当前搜索能力。

要求：

```txt
1. 支持按文件名搜索。
2. 支持按正文搜索。
3. 搜索结果可加入组合区。
4. 不重复加入同一文件，或重复加入时提示确认。
```

### 7.8 Composer 验收标准

```txt
[ ] 可以打开 Composer 弹窗。
[ ] 可以把当前文件加入 Composer。
[ ] 可以从文件列表选择多个文件加入 Composer。
[ ] 可以调整组合顺序。
[ ] 可以移除某个片段。
[ ] 可以清空组合区。
[ ] 组合预览正确。
[ ] 复制组合结果正确。
[ ] 导出组合结果为 .md 正确。
[ ] 保存组合结果为新的 .md 文件正确。
[ ] 组合结果中有 {{变量}} 时，可以使用模板变量填写表单。
[ ] Composer 不修改原始文件。
```
