MVP 定位非常清晰：

> "一个 PySide6 本地桌面置顶提示词文件夹工具。"

边界控制得当，明确排除了数据库、后端、AI 能力、Web UI 等。这降低了开发复杂度，缩短了验证周期。

**缺失/不明确项**：

1. **删除提示词的二次确认**：Plan 中仅要求"删除分类前必须二次确认"（6. 分类功能），但 7. 提示词文件功能中未明确要求删除提示词时二次确认。建议统一增加确认对话框，避免误删。
2. **未保存内容提醒的具体交互**：8.1 编辑模式提到"切换文件前，如果当前内容未保存，需要提醒用户保存或放弃修改"，但未说明 UI 形式（模态对话框 vs 非阻塞提示）。建议明确为模态确认对话框。
3. **新建提示词的默认分类**：Plan 未说明新建提示词时默认选中哪个分类。建议默认选中当前选中的分类。
4. **窗口状态持久化**：Plan 未提及是否记住窗口位置/大小。建议配置文件增加 `window_x`, `window_y` 字段。
5. **空状态处理**：Plan 未说明当 data/ 目录为空或不存在时的 UI 表现。建议首次启动时自动创建 data/ 目录并显示空状态提示。

**发现的问题**：

- **最终验收流程第 13 步**（"粘贴到 ChatGPT 网页版，内容为 Markdown 原文"）依赖于剪贴板内容的格式。如果用户复制的是 Markdown 源码，直接粘贴到 ChatGPT 是可行的，但需要注意 Windows 剪贴板可能携带富文本格式（RTF/HTML）。需要确保复制时仅写入纯文本（`text/plain`），避免粘贴到网页时出现格式异常。

## 3. 技术栈可行性分析

### 3.1 PySide6 桌面能力

#### 窗口置顶

| 方案 | 推荐度 | 说明 |
|------|--------|------|
| `Qt.WindowStaysOnTopHint` | ⭐⭐⭐ 首选 | 原生支持，Windows 上稳定 |
| `pywin32 SetWindowPos` | ⭐⭐ 备选 | 更底层，Qt 方案失效时兜底 |

**关键注意点**：
- 使用 `setWindowFlag(Qt.WindowType.WindowStaysOnTopHint, True)` 而非 `setWindowFlags()`（复数形式会覆盖其他标志）。
- 修改标志后必须重新调用 `show()` 生效。
- Windows 全屏游戏/视频上可能被 DWM 策略覆盖（Windows 行为，非 bug）。

#### 系统托盘

| 方案 | 推荐度 | 说明 |
|------|--------|------|
| `QSystemTrayIcon` | ⭐⭐⭐ 首选 | Qt 官方支持，Windows 11 兼容 |
| `pystray` | ⭐⭐ 备选 | 纯 Python，但增加依赖 |

**已知风险与缓解**：

| 风险 | 严重程度 | 影响版本 | 缓解措施 |
|------|----------|----------|----------|
| `hide()` 后 `show()` 不工作 | 中 | PySide6 ≤ 6.5.3 | **升级至 6.7+** |
| 托盘图标进程残留 | 低 | 所有 Windows 版本 | 应用正常退出时调用 `deleteLater()`，残留为 Windows Explorer bug，可接受 |
| 消息气泡被系统策略拦截 | 低 | 企业环境 | 不依赖气泡作为唯一信息渠道，配合状态栏提示 |
| 菜单引用丢失 | 低 | 所有版本 | Python 侧保持菜单引用：`self._tray_menu = QMenu(self)` |
| Win11 中键点击识别错误 | 低 | Win11 | 如需中键功能，用左键+右键 workaround |

**结论**：系统托盘功能可行，建议 PySide6 版本 ≥ 6.7。

### 3.2 全局快捷键

| 方案 | 维护状态 | 推荐度 | 优势 | 劣势 |
|------|----------|--------|------|------|
| **pynput** | ✅ 活跃（v1.8.1, 2025-03） | ⭐⭐⭐ 首选 | 跨平台、文档完善、作者响应快 | 打包需处理隐藏导入 |
| keyboard | ❌ 已归档（unmaintained） | ⭐ 不推荐 | API 极简 | 418 个 open issues 未处理，官方声明不再维护 |
| pywin32 | ✅ 活跃 | ⭐⭐ 备选 | Windows 原生 API，最稳定 | 需自行管理消息循环，代码冗长 |
| QShortcut | ✅ Qt 内置 | ❌ 不适用 | 简单 | 仅应用内快捷键，无法全局 |

**推荐方案：pynput**

```python
# 示例集成方式
from pynput import keyboard
from PySide6.QtCore import QObject, Signal

class GlobalHotkeyManager(QObject):
    toggle_requested = Signal()
    
    def __init__(self):
        super().__init__()
        self.hotkey = keyboard.GlobalHotKeys({
            '<ctrl>+<alt>+p': self.on_hotkey
        })
    
    def on_hotkey(self):
        self.toggle_requested.emit()
    
    def start(self):
        self.hotkey.start()
    
    def stop(self):
        self.hotkey.stop()
```

**主要风险**：

1. **UAC 权限边界**：如果用户在任务管理器、UAC 弹窗等 elevated 窗口中使用快捷键，pynput 需要以管理员权限运行才能捕获。这是 Windows 安全模型的限制，任何方案都一样。
   - **缓解**：MVP 阶段无需处理。如需覆盖 elevated 窗口，需在 manifest 中请求管理员权限。

2. **打包隐藏导入**：pynput 使用 `importlib` 动态加载后端模块，PyInstaller/Nuitka 可能遗漏。
   - **缓解**：打包时显式包含 `pynput.keyboard._win32` 和 `pynput.mouse._win32`。

3. **快捷键冲突**：`Ctrl+Alt+P` 可能与 IDE（如 VS Code 的 "显示所有命令" 默认快捷键冲突取决于插件）、浏览器插件冲突。
   - **缓解**：配置文件支持自定义快捷键；首次启动时检测冲突并提示用户。

### 3.3 Markdown 渲染方案

#### 方案对比

| 指标 | QTextBrowser 原生 | QWebEngineView | Markdown→HTML→QTextBrowser（推荐） |
|------|-------------------|----------------|-------------------------------------|
| 打包体积增量 | 0 | **~200MB** | ~5-10MB |
| GFM 表格支持 | ✅ | ✅ | ✅ |
| 代码语法高亮 | ❌ | ✅ | ✅（Pygments） |
| 数学公式 | ❌ | ✅ | ❌ |
| Mermaid 图表 | ❌ | ✅ | ❌ |
| 打包复杂度 | 低 | **高**（需处理 WebEngine 辅助进程） | 低 |
| 开发复杂度 | 低 | 中（需 Chromium 调试） | 低 |

#### QTextBrowser 原生 Markdown 支持边界

Qt 6 的 `QTextDocument::setMarkdown()` 支持 `MarkdownDialectGitHub`（默认），具体包括：

**✅ 支持**：标题、列表、代码块、引用、粗体、斜体、删除线、下划线、表格、任务列表、自动链接、Front matter、行内代码、图片

**❌ 不支持**：脚注、数学公式、Mermaid 图表、代码语法高亮

**⚠️ 限制**：`toMarkdown()` 存在已知问题，即使尝试禁用 GitHub 特性，输出仍可能包含它们。

#### 推荐方案：Markdown → HTML → QTextBrowser

**架构**：

```
Markdown 源码
    ↓
python-markdown / markdown-it-py → HTML
    ↓
Pygments → 带 CSS 的语法高亮 HTML
    ↓
QTextBrowser.setHtml(rendered_html)
```

**推荐库组合**：
- `markdown` (~100KB) 或 `markdown-it-py` (~100KB)：核心解析器
- `Pygments` (~5MB)：代码语法高亮，支持 500+ 语言

**为什么不选 QWebEngineView？**
- 体积增加 200MB，对于 MVP 过于沉重
- 打包复杂度高（需处理 QtWebEngineProcess）
- MVP 不需要 Mermaid/数学公式等高级渲染

**编辑模式推荐**：`QPlainTextEdit` + 自定义 `QSyntaxHighlighter`

理由：
1. 编辑的是 Markdown 源码，不是 WYSIWYG，纯文本编辑器更合适
2. 性能优于 `QTextEdit`，可处理大文档
3. 行号显示更自然
4. 社区验证：知名 Qt Markdown 编辑器 `qmarkdowntextedit` 基于 `QPlainTextEdit`

### 3.4 打包分发

| 工具 | 构建速度 | 启动速度 | 体积 | 杀软误报 | 推荐度 |
|------|----------|----------|------|----------|--------|
| **PyInstaller (onedir)** | ⚡ 快 | ⚡ 快 | 中 | 中 | ⭐⭐⭐ 首选 |
| PyInstaller (onefile) | ⚡ 快 | 🐌 慢（解压到临时目录） | 略小 | 中 | ⭐⭐ |
| Nuitka | 🐌 慢 | ⚡ 快 | onefile 略小 | 低 | ⭐⭐ 备选 |
| cx_Freeze | 🐌 较慢 | ⚡ 快 | 较大 | 中 | ⭐⭐ |

**推荐方案：PyInstaller onedir 模式**

- 生态最成熟，对 PySide6 支持最好
- onedir 启动接近原生，避免 onefile 的解压延迟
- 基础 PySide6 应用约 20-30MB，可接受

**主要风险**：
- **杀毒软件误报**：所有 Python 打包工具的通病。可通过代码签名证书、向杀软厂商提交白名单缓解。
- **Qt 平台插件路径**：打包时需确保 `platforms/` 目录正确包含，否则运行时报错。

## 4. 识别的风险与缓解措施

### 4.1 高风险

| 风险 | 影响 | 发生概率 | 缓解措施 |
|------|------|----------|----------|
| **快捷键冲突** | 用户无法呼出应用 | 中 | 支持自定义快捷键；首次启动检测冲突并提示 |
| **杀软误报导致无法运行** | 用户无法启动应用 | 中 | 使用 PyInstaller onedir；考虑代码签名；提供安装说明 |
| **文件编码问题（中文路径）** | 文件读写失败 | 中 | 所有文件操作使用 UTF-8 编码；路径处理使用 `pathlib.Path` |

### 4.2 中风险

| 风险 | 影响 | 发生概率 | 缓解措施 |
|------|------|----------|----------|
| **外部编辑器修改文件后的同步** | 应用显示旧内容 | 中 | 使用 `QFileSystemWatcher` 监听 data/ 目录变化，自动刷新 |
| **未保存内容丢失** | 用户编辑内容丢失 | 中 | 切换文件/关闭窗口前强制确认保存；增加"已修改"状态指示 |
| **pynput 打包后失效** | 全局快捷键不工作 | 低 | 打包时显式包含 `pynput.keyboard._win32` 等隐藏导入 |
| **UAC  elevated 窗口快捷键失效** | 在任务管理器等窗口中快捷键不响应 | 低 | MVP 阶段不处理；如需覆盖，请求管理员权限运行 |

### 4.3 低风险

| 风险 | 影响 | 发生概率 | 缓解措施 |
|------|------|----------|----------|
| **托盘图标残留** | 应用退出后图标仍在 | 低 | Windows Explorer bug，鼠标悬停后消失，可接受 |
| **QTextBrowser Markdown 渲染差异** | 渲染效果与用户预期不符 | 低 | 明确文档说明支持的 Markdown 语法子集 |
| **窗口置顶被全屏应用覆盖** | 游戏/视频时窗口不置顶 | 低 | Windows DWM 策略，非应用 bug，可接受 |

## 5. 改进建议

### 5.1 Plan 内容补充（建议修改 prompt_anywhere_mvp_plan.md）

1. **增加"删除提示词二次确认"要求**
   - 位置：7. 提示词文件功能
   - 建议：在删除提示词前增加模态确认对话框，与分类删除保持一致。

2. **明确"未保存提醒"的 UI 形式**
   - 位置：8.1 编辑模式
   - 建议：明确为模态对话框，提供"保存"、"不保存"、"取消"三个选项。

3. **增加空状态处理说明**
   - 位置：2. 主界面设计 / 6. 分类功能
   - 建议：首次启动时自动创建 `data/` 目录；空分类时显示"暂无提示词，点击 + 新建"；空 data 目录时显示引导提示。

4. **增加窗口状态持久化**
   - 位置：13. 配置文件
   - 建议：配置文件增加 `window_x`, `window_y`, `window_width`, `window_height` 字段，记住用户调整的窗口状态。

5. **明确新建提示词的默认分类**
   - 位置：7. 提示词文件功能
   - 建议：新建提示词时，默认选中当前左侧选中的分类。

6. **增加剪贴板纯文本格式要求**
   - 位置：8.3 复制规则 / 10. 复制功能
   - 建议：明确复制时仅写入 `text/plain` MIME 类型，避免携带富文本格式导致粘贴到网页出现异常。

### 5.2 技术实现建议

1. **文件监控自动刷新**
   - 使用 `QFileSystemWatcher` 监听 `data/` 目录，当外部编辑器修改文件时自动刷新内容，避免用户困惑。

2. **配置热重载**
   - 监听 `config.yaml` 变化，支持不重启应用修改快捷键等配置。

3. **快捷键自定义**
   - 配置文件支持 `hotkey` 字段自定义，首次启动时若检测到冲突则提示用户。

4. **PySide6 版本要求**
   - 建议明确依赖 PySide6 ≥ 6.7，以规避系统托盘的已知 bug。

## 6. MVP 阶段推荐决策

### 6.1 技术选型决策表

| 功能 | 推荐技术 | 关键注意点 |
|------|----------|------------|
| 桌面窗口 | PySide6 ≥ 6.7 | `setWindowFlag()` 避免覆盖标志 |
| 窗口置顶 | `Qt.WindowStaysOnTopHint` | 修改后重新 `show()` |
| 系统托盘 | `QSystemTrayIcon` | 保持菜单引用，升级 6.7+ |
| 全局快捷键 | **pynput** | 打包时加入隐藏导入 |
| 编辑模式 | `QPlainTextEdit` + `QSyntaxHighlighter` | 实现 Markdown 语法高亮 |
| 渲染模式 | `QTextBrowser` + `python-markdown` + `Pygments` | `setHtml()` 显示带语法高亮的 HTML |
| 文件监控 | `QFileSystemWatcher` | 监听 `data/` 目录自动刷新 |
| 配置文件 | PyYAML | UTF-8 编码，支持热重载 |
| 打包分发 | **PyInstaller (onedir)** | onedir 启动更快；准备应对杀软误报 |

### 6.2 开发顺序建议

沿用 Plan 的 Phase 划分，但建议调整：

1. **Phase 1：桌面壳子**（2-3 天）
   - 主窗口、置顶、托盘、全局快捷键
   - 先验证核心体验是否流畅

2. **Phase 2：本地文件读取**（1-2 天）
   - 读取 data/ 目录，三栏 UI，点击加载内容
   - 同时集成 `QFileSystemWatcher`

3. **Phase 3：CRUD 操作**（2-3 天）
   - 分类和提示词的增删改重命名
   - 同时加入删除二次确认、未保存提醒

4. **Phase 4：Markdown 双模式**（2-3 天）
   - 编辑模式（QPlainTextEdit + 语法高亮）
   - 渲染模式（python-markdown + Pygments + QTextBrowser）

5. **Phase 5：搜索/复制/导出**（1-2 天）
   - 关键词搜索、剪贴板复制、文件导出

**预计总工期**：**1-2 周**（单人全职开发）

### 6.3 是否需要调整 MVP 边界？

**不需要**。当前 MVP 边界合理，以下功能建议严格保持在 v1.1+ 再考虑：

- ❌ 数据库 / 后端服务
- ❌ AI 能力（自动分类/标签/总结）
- ❌ 语义搜索 / 向量检索
- ❌ 云同步 / 多用户
- ❌ 浏览器插件
- ✅ v1.1 可考虑：Mermaid 图表渲染（如需，再评估 QWebEngineView）
- ✅ v1.1 可考虑：更丰富的 Markdown 语法高亮主题
